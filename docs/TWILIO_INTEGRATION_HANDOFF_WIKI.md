# Twilio Integration Handoff Wiki

Last updated: 2026-07-05

## 1. Purpose

This page is the operational handoff for the Twilio + CIF v2 integration in this repository.

Use this document to:

- Understand end-to-end behavior from Twilio events to Dynamics CIF sessions.
- Configure prerequisites in Dynamics and Twilio.
- Validate the integration after deployment.
- Troubleshoot the most common production and UAT issues.

## 2. Scope and Context

This project is a React-based CIF channel provider that:

- Connects to Twilio Conversations as an agent.
- Creates and manages CIF sessions in Dynamics.
- Raises HIA conversation events through CIF (`onConversationLoad`, `onCustomerMessageReceived`).

The widget acts as the bridge between:

- Twilio Conversations (chat transport)
- CIF v2 / Dynamics (session + notification framework)
- HIA event consumers (intent/recommendation workflows)

## 5. Dynamics / CIF Prerequisites

The integration assumes CIF is available in the hosting Dynamics context.

Required CIF artifacts:

- Session templates:
  - `msdyn_twilio_dashboard` (dashboard shell)
  - `msdyn_3p_session` (one session per accepted customer conversation)
- Notification template:
  - Default selected from available notification templates during bootstrap.

Bootstrap behavior:

1. Loads CIF library via base URL parameter.
2. Loads session/app/notification templates into GlobalStore.
3. Launches provider by creating dashboard session (`msdyn_twilio_dashboard`).

## 6. Runtime Behavior and Event Flow

### 6.1 Startup

1. App initializes `Bootstrapper`.
2. CIF templates are discovered and cached.
3. A dashboard CIF session is created.
4. Twilio client is initialized using token endpoint and selected agent identity.

### 6.2 Incoming Conversation

When Twilio emits `conversationJoined`:

1. Widget raises CIF notification (`notifyEvent`).
2. If agent accepts:
   - Creates new CIF session with template `msdyn_3p_session`.
   - Stores bidirectional mapping between CIF session ID and Twilio conversation SID.
3. Conversation loads when CIF focus switches to that session.

### 6.3 Session Awareness

The panel listens for CIF events:

- `onSessionSwitched`: load corresponding Twilio conversation if mapped; otherwise show dashboard.
- `onSessionClosed`: clean session↔conversation mappings and allow future re-raise of conversation load.

### 6.4 HIA Event Raising

Raised by `HIAConversationEventHelper` through `CIFV2.raiseEvent`:

- `onConversationLoad`
  - Raised once per conversation lifecycle in the widget (guarded by loaded conversation set).
- `onCustomerMessageReceived`
  - Raised for inbound non-agent messages only.
  - Includes message ID and optional timestamp.

Contract details are documented in `docs/HIA_CONVERSATION_EVENTS.md`.

## 7. Session and Conversation Mapping Model

Two in-memory maps are used in `TwilioAgentPanel.tsx`:

- `sessionToConversationMap`: CIF session ID -> Twilio conversation SID
- `conversationToSessionMap`: Twilio conversation SID -> CIF session ID

Why this matters:

- Enables restoring the correct conversation on CIF focus changes.
- Avoids duplicate session creation for already-mapped conversations.
- Supports session cleanup and idempotent re-entry behavior.

## 9. Validation Checklist (UAT/Prod Smoke)

Use this checklist after any deployment:

1. Dashboard session opens successfully in Dynamics.
2. Agent can authenticate and Twilio client initializes.
3. New Twilio conversation triggers Dynamics notification.
4. Accept action creates new `msdyn_3p_session`.
5. Session switch loads correct Twilio conversation.
6. Customer message raises `onCustomerMessageReceived` event.
7. Agent-authored message does not raise customer-message event.
8. Session close removes map entries and allows clean re-entry.

## 10. Known Risks and Gaps

1. Hardcoded Twilio auth/user endpoints in UI code.
2. Intent family ID currently static in panel state.
3. Mapping state is in-memory only (no durable persistence across full reloads).
4. Error handling is mostly console-driven; no centralized telemetry envelope in this repo.

## 11. Recommended Next Improvements
